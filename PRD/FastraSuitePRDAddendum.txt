1. Introduction & Purpose
This document serves as a formal addendum to the existing FastraSuite Product Requirements Specification. It defines the functional requirements for two newly introduced modules:
* Project Costing Module: the central financial tracking and reporting hub for all project-related spending, powered by the Project Costing Engine.
* Project Request Module: a mobile-first request initiation layer designed specifically for site workers and field teams.
These modules complement the existing Purchase, Inventory, and Invoice modules and complete the core operational loop of FastraSuite.
1.1 Core System Philosophy
FastraSuite is designed to simplify ERP complexity by clearly separating operational responsibilities across four layers:
Layer
	Actor
	Responsibility

	The system must enforce strict alignment between what is requested on-site, what is approved within the project budget, and what is ultimately paid by finance. All spending must originate from validated site activity. Every transaction must be traceable to a project, a WBS element, and a cost category. Payments are only made for verified and validated work or deliveries.
1.2 System Architecture Note
FastraSuite operates as a multi-tenant SaaS platform. Each registered company exists as an isolated tenant with its own data environment. No data is shared across companies. All companies share the same underlying codebase and infrastructure.
1.3 Subscription Model
FastraSuite is offered on two subscription tiers. The Core tier includes four modules: Project Request Module, Project Costing Module, Invoice Module, and Inventory Module. The Enterprise tier includes all core modules plus Sales (CRM, Quotation, SO), Purchase, HR, and Asset Management. Companies on the Core tier can upgrade to Enterprise at any time. All data created during the Core subscription is retained and carried over upon upgrade. The Purchase Order feature lives inside the Invoice Module on Core and moves to the Purchase Module on upgrade without any data loss.

3. Module Overview & Cross-Cutting Features
3.1 Project Request Module
The Project Request Module is the mandatory entry point for all site activity and spending requests. It is a streamlined, mobile-optimized interface designed for site workers and field teams. Every spending request must be tagged with a WBS element and a cost code and is automatically validated by the budget validation gate before reaching any human approver. This module is strictly for initiating requests and routing approvals — it does not process payments.
This module covers seven request types:
1. Purchase Request
2. Labour Request
3. Petty Cash Request
4. Subcontractor Request — supports both lump-sum and milestone-based payment terms
5. Plant & Equipment Request
6. Material Consumption Request
3.2 Project Module
The Project Module is the financial management and reporting backbone of FastraSuite, powered by the Project Costing Engine—an active validation and tracking engine, not a passive reporting tool. It manages the work breakdown structure, holds the cost code library, runs the budget validation gate on every request submission, tracks committed and actual amounts in real time, and displays financial status via the BvA Dashboard.

3.9 Work Breakdown Structure (WBS) Manager
The work breakdown structure is a hierarchical breakdown of a project into levels. For example, phases, sub-phases, and tasks. Every budget line, request, and actual cost in FastraSuite must be tagged to a specific WBS element. This enables financial performance tracking at any level of the project hierarchy, from the overall project down to an individual task.
FastraSuite uses one unified WBS structure that works for all industries. When a company registers, the system pre-loads a standard template with default level names. The Company Admin can rename these levels, adjust the number of levels, and customize the structure to match their organization’s terminology. No industry requires a different underlying system — only the labels differ.
 
WBS Structure Rules:
Rule
	Detail
	Configurable depth
	The number of hierarchy levels is configurable by the company admin. There is no fixed limit set by the system
	Customizable level names
	All WBS level names can be renamed by the company admin to match the organization’s terminology. For example, Level 1 may be renamed Phase, Programme, or Well
	System-generated IDs
	The system assigns a unique ID to each WBS element on creation. This ID cannot be manually edited after the first transaction is attached to that element
	Leaf-only budget attachment
	Only leaf elements — those with no children — can have budget lines attached. The system blocks budget attachment to any parent element.
	Parent total rollup
	Parent element totals are always calculated automatically from the sum of their children. They are never manually entered at the parent level
	No deletion after transaction
	A WBS element cannot be deleted after any transaction has been attached to it. It can only be marked as "Cancelled."
	Cancelled elements
	A Cancelled WBS element remains visible in history and reports but cannot receive any new transactions
	New leaf elements
	New leaf elements can be added to an Active project without triggering a full budget revision

3.11 Budget Validation Gate
The Budget Validation Gate is an automated check that fires every time a request is submitted. It checks whether the requested amount is within the available budget for the specific WBS element and cost code combination tagged on the request. It fires before any human approver sees the request.
 
Available Budget Formula:
Available Budget = Budgeted Amount − Actual Amount − Committed Amount
Term
	Definition
	Budgeted Amount
	The approved budget for a specific WBS element and cost code combination
	Actual Amount
	Confirmed payments already made against that WBS element and cost code
	Committed Amount
	Approved but not yet paid requests against that WBS element and cost code
	Available Budget
	What remains available for new spending on that specific line
	Critical Rule: Committed amounts are always deducted from the available budget in every validation check. A request that has been approved but not yet paid still consumes the available budget.
Pre-Submission Budget Visibility:
Before submitting any request, the site worker can see the available budget for the selected WBS element and cost code combination. The system displays the available budget amount only — the worker does not see the full breakdown of budgeted, actual, or committed amounts separately. This gives the worker enough information to self-check their request before submission without exposing sensitive financial details.
Display Rule
	Detail
	What is shown
	Available Budget amount for the tagged WBS element and cost code—updates in real time as the worker selects both fields
	When it appears
	Immediately after both WBS Element and Cost Code are selected — before the worker enters the requested amount
	Format
	Displayed as: 'Available Budget: ₦[amount]' — prominently positioned above the amount or cost fields
	What is NOT shown
	Budgeted Amount, Actual Amount, Committed Amount — these are never shown to site workers
	Insufficient budget indicator
	If the worker enters an amount that exceeds the Available Budget, a warning is shown before submission: 'This request exceeds the available budget and will require a budget review'
	 
Validation Gate Flow:
Step
	Action
	Result
	1
	Site worker selects WBS Element and Cost Code
	System displays Available Budget amount for that combination
	2
	Worker completes the form and taps Submit
	System intercepts the request before routing to any approver
	3
	System recalculates Available Budget at the exact moment of submission
	Budgeted − Actual − Committed for that WBS + Cost Code
	4a
	Within budget
	Request proceeds to the PM approval queue normally
	4b
	Over budget
	Request is flagged and held in the Overrun Queue — PM is notified
	5
	Worker sees the result
	Within budget: standard confirmation. Over budget: 'This request requires a budget review before it can be processed. You will be notified of the outcome.'
	 
Overrun Queue — PM Actions:
When a request is held in the Overrun Queue, the PM has three options:
Action
	What It Does
	Audit Log Entry
	Approve with Override
	Request is approved despite the overrun — PM must enter a mandatory note explaining the override
	Permanent immutable entry: PM name, timestamp, override note, overrun amount
	Reject
	Request is rejected — committed amount is immediately released back to Available Budget — submitter is notified with the reason
	Permanent immutable entry: PM name, timestamp, rejection reason
	Reallocate then Approve
	PM moves budget from another WBS element and cost code line to cover the overrun — then approves the request
	Permanent immutable entry: PM name, timestamp, source line, destination line, amount reallocated
	All three PM actions generate a permanent audit log entry that cannot be edited or deleted by any user at any level.

	Site Worker — over budget

	PM — overrun notification
	Full breakdown: WBS element, Cost Code, Budgeted Amount, Actual Amount, Committed Amount, Available Budget, Requested Amount, Overrun Amount

3.12 Overrun Alert System
The Overrun Alert System monitors budget consumption across all WBS elements and cost codes on every active project. When consumption reaches a configured threshold, the system sends an alert to the designated recipients. This is a proactive warning system — it fires before the budget is exhausted, giving the PM time to act.
Alert Threshold Formula:
* Alert fires when: (Actual Amount + Committed Amount) ÷ Budgeted Amount × 100 ≥ Threshold %
Alert Threshold Configuration:
Setting
	Detail
	Company default threshold
	A single default threshold percentage set at company level — applies to all projects and all cost categories unless overridden
	Project-level override
	The PM can override the company default threshold for each individual project
	Category-level override
	The PM can set different thresholds per cost category within the same project — for example, Labour at 75% and Materials at 85%
	Alert recipients
	Configurable per project — default recipients are the PM and Finance
	 
Alert Behaviour:
1. Alert fires once when the threshold is first crossed — it does not fire repeatedly on every subsequent transaction
2. Alert resets if budget is reallocated and consumption drops below the threshold — it will fire again the next time the threshold is crossed

6.3 Integration Rules
•         A request in the Project Request Module must always be linked to an active project — closed projects cannot receive new requests
•         Every transaction across all modules must carry a valid WBS element reference and a cost code reference — no transaction can be submitted without both fields
•         The WBS element referenced must be a leaf element — parent elements cannot receive transactions
•         The Budget Validation Gate fires on every request submission across all modules — Inventory, and Project Request — without exception
•         When a request is approved, the system automatically creates the corresponding record in the target back-office module in draft status — carrying the WBS element and cost code references
•         The WBS element and cost code references must carry through from the originating request to every downstream record — Project Request, PO, Invoice etc.
•         Deletion of a back-office record that originated from a Project Request must trigger a reversal of the corresponding committed amount in the Project Costing Module
•         Payment cancellation or reversal in the Invoice Module must trigger a reversal of the actual cost entry in the Project Costing Module
•         Finance must be automatically notified whenever a Material Receipt quantity does not match the corresponding PO quantity

16. Glossary
Term
	Definition
	Actual Amount
	Total confirmed payments made against a specific WBS element and cost code combination — sourced from the Invoice Module via the Finance Ledger Extension
	Available Budget
	The budget remaining for new spending on a specific WBS element and cost code line. Formula: Budgeted Amount minus Actual Amount minus Committed Amount
	Back-Office Module
	The internal processing modules — Purchase, Inventory, and Invoice
	BvA Dashboard
	Budget vs Actual Dashboard — the real-time financial performance view showing Budgeted, Committed, Actual, Remaining, and % Consumed for every WBS element and cost code combination on a project
	Budget Health
	A system-calculated indicator for each WBS element and cost code line — On Track, At Risk, or Over Budget — based on configured alert thresholds
	Budget Revision
	A formal request to change the approved project budget on an active project — subject to an approval workflow — every revision is logged in the audit trail
	Budget Template
	The column structure used to define and upload a project budget. FastraSuite uses one unified template that is customisable by the Company Admin — additional columns can be added and existing columns can be renamed to match the company's terminology
	Budget Validation Gate
	The automated check that fires on every request submission across all modules. It checks the Available Budget for the tagged WBS element and cost code combination and either allows the request to proceed or holds it in the Overrun Queue
	Committed Amount
	The total value of approved but not yet paid requests and purchase orders against a specific WBS element and cost code combination. Committed amounts are always deducted from Available Budget in all validation checks
	Committed to Actual
	The conversion that occurs when a committed approved request or order is paid via the Invoice Module and confirmed through the Payment Approval Workflow
	Cost Code
	A spending category from the company-level master list. Every transaction must be tagged with a cost code — for example, Labour, Materials, or Plant & Equipment
	Cost Code Library
	The company-level master list of all available cost codes. Projects select from this library — they cannot create their own project-specific codes
	Finance Ledger Extension
	The extension to the existing Invoice Module that adds 3-Way Match logic, Accounts Payable Queue, Cash Flow View, and Payment Approval Workflow
	Inventory Ledger
	A complete chronological record of all stock movements at a site — including timestamp, user, movement type, quantity change, running balance, project, WBS element, and source document reference
	Leaf Element
	A WBS node that has no children. Only leaf elements can have budget lines attached. Parent element totals roll up automatically from their children.
	Material Receipt Logging
	The on-site confirmation by a Site Supervisor of the actual quantity of goods received against a Purchase Order — provides the site receipt data point for the 3-Way Match process
	Milestone-Based Payment
	A payment structure for subcontractor requests where the total contract value is divided into defined milestones — each milestone payment is only released when the PM marks the corresponding milestone as complete
	Overrun Queue
	The holding queue for requests that have exceeded the Available Budget for their tagged WBS element and cost code combination — the PM reviews and resolves all held requests
	Project Budget
	The formally approved total spending limit for a project — broken down by WBS element and cost code
	Project Costing Engine
	The financial control system of FastraSuite — manages the WBS structure, cost code library, budget setup, budget validation, committed and actual tracking, overrun alerts, and the BvA Dashboard
	Project Request
	A field-initiated request submitted via the Project Request Module for goods, labour, cash, services, or material recording
	Remaining Budget
	Project Budget minus total Committed Amount minus total Actual Amount
	3-Way Match
	The automated verification that checks three data points before an invoice is approved for payment: the Purchase Order amount, the site receipt quantity confirmed by the Site Supervisor via Material Receipt Logging, and the vendor invoice amount
	Variance
	The difference between the Project Budget and the Actual Amount spent to date
	WBS
	Work Breakdown Structure — the hierarchical breakdown of a project into levels. The Company Admin defines the number of levels and the name of each level. Every budget line and transaction must be tagged to a specific WBS element.
	WBS Element
	A single node in the Work Breakdown Structure — for example, a phase, sub-phase, or task. Only leaf elements (those with no children) can receive budget lines and transactions.
